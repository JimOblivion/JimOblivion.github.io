SubPath的使用

使用SubPath的原因

  1. 查看容器 kubectl -n zy-test get po

  2. 进入容器 kubectl -n zy-test exec -it nginx-deploy-7b49587448-vqb7l -- sh

  3. 找到容器中文件/etc/nginx/conf.d/nginx.conf

  4. 拷贝内容，新建文件到我们的目录中nginx.conf文件中

  5. 创建configMap
    kubectl -n zy-test create configmap nginx-conf-cm --from-file=./nginx.conf

  6. 编辑deploy，新增数据卷
    kubectl -n zy-test edit deploy nginx-deploy

  7. 查看容器状态，发现报错 kubectl -n zy-test get po

  8. 描述容器中的错误内容 kubectl -n zy-test describe po nginx-deploy-76459d6d66-fxmsg

  9. 进入容器查看文件是否加载，发现报错，原因是我们容器未启动成功。
    解决办法: 给容器加一个睡眠时间，让他不要挂掉。

  10. 编辑deployment kubectl -n zy-test edit deploy nginx-deploy

  11. 查看当前pod

  12. 进入容器 kubectl -n zy-test exec -it nginx-deploy-5f986958cc-fh8wf -- bash

    发现容器目录中其他文件都没有了，只剩下nginx.conf。原因是使用ConfigMap或Secret挂载到目录的时候，会将容器中源目录覆盖掉，此时我们可能只想覆盖目录中某一个文件， 所以要使用subpath。

使用SubPath

  • 编辑deploy, ==#1== 和 ==#2==的路径要保持一致

  • 查看pod的运行状态

  • 进入容器内部，文件正常

ConfigMap热更新

  • 如果更新configmap中的配置，会不会更新到pod中呢？

  • 修改cm

  • 进入容器内部查看配置已经变更

    使用replace替换，结合--dry-run参数。该参数的意思是打印yaml文件，但是不会将文件发送给apiServer，结合-oyaml输出yaml文件就可以得到一个配置好但是没有发给apiserver的文件，然后再结合replace监听控制台输出得到yaml数据即可实现替换。

  • 查看确认当前配置

  • 把前置作为输入

  • 执行完成之后发现配置已经被替换为新的

不可变配置

对于一些敏感服务的配置文件，在线上有时是不允许修改的，此时只需要再配置configMap的时候设置immutable:true即可

  • 编辑configMap， 打开配置项

  • 再次编辑配置项，修改属性值，此时已经不允许修改
